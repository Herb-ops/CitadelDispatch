# Issue Construction Rules

## Fixed structure

Each issue must contain:

1. **Title**
2. **One-sentence abstract**
3. **Body assembled from relevant modules**

No fixed section order is required beyond readability.

## Assembly logic

1. Start with the highest-consequence signal.
2. Add only modules supported by meaningful changes.
3. Prefer depth on 1–3 important developments over shallow coverage of many items.
4. Include disagreement framing where legitimate camps diverge.
5. End with clear primary sources and a forward watchlist.

## Inclusion criteria (publish if at least one is true)

- Meaningful protocol/policy direction shift
- Important implementation/release change in key repos
- Material network-level behavior signal with defensible evidence
- Coordination or governance conflict with ecosystem relevance
- New explanatory context needed to understand active disputes

## Exclusion criteria

- Recycled talking points without new evidence
- Pure personality conflict without technical/governance relevance
- Commentary based on second-hand social media summaries
- Price-only narratives disconnected from protocol realities

## Style constraints

- Neutral language; avoid loaded framing
- Explicitly separate facts, interpretations, and open questions
- Attribute claims to sources, not “the community” abstractions
- Keep arguments falsifiable with links and context
